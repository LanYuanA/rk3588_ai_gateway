# RK3588 多路边缘 AI 视频处理网关项目指南

本文档用于记录和拆解基于 RK3588 的多路流融合检测项目的整体框架和代码演进过程。随着项目的推进，本文档将持续更新。

## 一、 项目背景与架构蓝图

本项目致力于充分利用 RK3588 强大的硬件加速能力（8核CPU + MPP VPU + RGA 2D加速 + 6TOPS NPU），完成“多路视频流接入 -> 硬件协处理解码 -> NPU AI 推理 -> 多画面融合拼接 -> 流媒体服务分发”的完整边缘计算节点(AI-NVR)能力。

**核心技术栈映射：**
- **多路流处理架构**：基于生产-消费者模型，解耦拉流、解码、推理、拼接、推流线程。
- **MPP/RGA/RKNN 实战**：硬件编解码 (MPP)、图像缩放/拼接 (RGA)、模型硬件推理 (RKNN)。
- **实时系统优化**：跳帧推理、复用、帧率控制、线程池并发。
- **检测结果融合**：多模型输出叠加。
- **工程稳定性**：优雅退出，防僵尸进程，自动断流重连。
- **推流协议链路**：RTSP / RTMP / GB28181 输出。

## 二、 阶段演进记录

### 阶段一：基础骨架与并发模型搭建 (当前完成)
在此阶段，我们使用现代 C++ (C++11/14) 搭建了整个应用的基础结构，主要处理以下两个问题：
1. **线程解耦（多路隔离）**：主程序拉起了针对 4 路视频分别独立的拉流和解码线程，并启动了全局的推理（AI检测）和推流（合成导出）线程。这些单独的线程为后续解耦硬件模块打下了地基，确保当一路由于网络断开时，不会卡死整个应用。
2. **生命周期管理与安全退出**：声明了全局原子变量 `g_system_running`。主线程在捕捉阻塞（或未来捕捉 SIGINT/Ctrl+C 信号）后，能向所有子线程发送退出广播，并通过 `thread::join()` 等待其释放资源。这是面试中极常考察的**“如何避免terminate导致硬件死锁抛错”**的核心处理方案。

**当前文件结构：**
- `CMakeLists.txt`: C++ 构建脚本，引入 Threads 依赖。
- `include/common.h`: 存放跨文件共享的全局变量以及配置。
- `src/main.cpp`: 程序的入口点，包含了对上面所述的各核心线程的模拟以及优雅退出逻辑的演示。

### 阶段二：生产者-消费者模型与缓冲队列 (当前完成)
在此阶段，我们实现了线程间数据安全流转的“大动脉”——线程安全的环形队列（Thread-Safe Queue）。在视频流处理应用中，由于各个环节处理速度不一致（比如网络拉流 30帧/秒，但如果模型太大，NPU 推理只有 15帧/秒），缺乏缓冲机制会导致：**画面剧烈卡顿、或者未处理完的帧堆积在内存中导致系统直接被 Linux OOM(Out of Memory) 杀掉**。
- **`ThreadSafeQueue` 的容错特性**：使用 `std::mutex` 和 `std::condition_variable` 实现同步。最关键的是加入了**自动丢弃旧帧（跳帧）机制**。当队列满了之后，再 push 进来的新视频帧，会把队头最旧的那一帧挤出去。这在直播、安防这种强调**实时性**的场景中至关重要（宁愿跳帧，也不要延迟）。
- **流程整合**：`main.cpp` 被更新，现在通过 `ThreadSafeQueue<VideoFrame>` 实例进行通讯。拉流线程（生产者）负责源源不断向队列里送入带有 `frame_id` 的结构体；NPU 线程（消费者）从 4 路流的队列里轮询提取视频帧进行“模拟推理”。

**当前文件结构新增：**
- `include/ThreadSafeQueue.h`: 封装了锁的条件安全的自丢弃式缓冲队列模板类。
- 更新了 `include/common.h`: 新增了 `VideoFrame` 视频帧结构体。

> **下一步计划**：引入真实的 OpenCV/FFmpeg 依赖，替换此时 `main.cpp` 里的“模拟拉流 `sleep_for(33ms)`”，拉取一个真正的网络摄像头 RTSP 地址，并获取真正的图片矩阵送入队列！

### 阶段三：引入真实媒体源与数据流打通 (当前完成)
在此阶段，我们通过引入工业界大名鼎鼎的 **OpenCV** (底层自动对接 FFmpeg 或 GStreamer)，将我们的框架从“空气模拟”变成了“拉取真实视频流并提取像素数据”。
- **`CMakeLists.txt` 构建系统强化**：通过加入 `find_package(OpenCV REQUIRED)` 自动管理系统环境中的动态链接库依赖。确保能够调用真实的视觉处理网关。
- **改写 `VideoFrame` 结构体**：我们让结构体增加了 `cv::Mat image`，实现了每一帧从像素解码到内存，并进入核心的安全自丢弃缓冲队列。这意味着我们的消费者 (NPU 队列) 拿到的是极具价值的、活生生的矩阵像素图，而且打印出了当前画面的尺寸数据。
- **对输入协议的包容性(容错)**: 重写后的 `streamPullerAndDecoderThread` 对于输入端实现了 `cv::VideoCapture cap(url)`，这代表了我们的程序目前已经具备同时吃下：本地 USB 摄像头 (`/dev/video0`)、本地视频文件 (`test.mp4`)、甚至是网络 IPC 摄像头的高配 `rtsp://IP:PORT/stream` 码流的能力。且添加了应对流意外中断时的循环重连逻辑。

> **下一步计划**：真正的 RKNN NPU 接管！我们将引入 `rknn_api.h`，绑定模型，替换掉“模拟推理”模块，使画面进入真正的网格目标检测阶段。同时讨论如果是零拷贝架构，我们如何把 OpenCV 的 `cv::Mat` 换成 RGA 高速图形硬件的 `dma_buf` 内存块管理。

### 阶段四：实战 RKNN C-API，点亮 NPU 加速处理引擎 (当前完成)
在此阶段，我们挥别了 CPU 或者 `sleep` 模拟计算，正式引入了瑞芯微专用的 NPU `librknnrt.so`。
我们把非常面向过程和充满指针调用的 C-API，封装成了一个优雅的 `RKNNDetector` C++ 类。
- **系统状态同步**：现在的流转管线变成了：`OpenCV 拉流 -> Image(cv::Mat) -> 无锁队列 -> 轮询弹出 -> NPU(rknn_inputs_set) -> NPU(rknn_run) -> 推理耗时打印`。
- **构建环境与硬件交互**：通过 CMake 连接上了 `rknnrt` 底层运行库动态链接。实现了将模型二进制 `yolov8n_face_fp.rknn` 直接写进硬件显存，并且进行像素格式转换 (BGR->RGB)、图像强求缩放，以满足深度学习算子（Tensor）的输入要求。

> **接下来的难点与下一步（阶段五）**：
> NPU 已经输出了（如：边框和置信度的浮点数组），但这些只是冷冰冰的一维数字。**我们需要写 YOLO 的后处理：反解锚框 (Decoding Bounding Boxes)，并且编写 NMS (非极大值抑制)** 去除多余的人脸框！完成后用 OpenCV 把框画到我们原本的视频流上！

### 阶段五：深度排爆与 YOLOv8 后处理开发 (当前完成)
在此阶段，我们将系统真正投入了物理开发板运行，并经历了工业界极具代表性的高难度 C++ 端硬件排坑，最终在 `yolov8_face_fp.rknn` 模型上实现了完美的 **NMS 后处理与画面实时人脸框绘制**。
- **深度解决 NPU 崩溃与异步跳帧**：解决了由于多核抢占、浮点模型 (`_fp`) 在 `uint8` 隐式转换时的 NaN 溢出以及 C++ 过高并发引起的 DMA 挂起 (`failed to submit`) 等致命 Bug。我们将这些过程梳理成了一份 `RK3588_NPU_DEBUG_REPORT.md` 复盘报告。
- **YOLOv8 后处理与 NMS (非极大值抑制)**：完全重写了 `rknn_detector.cpp` 中的 `inference()`。通过读取 `output_attrs[0].dims` 完全自适应了常见的 NCHW / NHWC 格式输出。提取模型跑出来的原始张量，从 1 维 Float 数组中成功解码出每张脸的 `cx, cy, w, h`，并且映射回了 `image` 处理前的原始高宽。同时引入了标准的 `IoU NMS` 框过滤算法。
- **验证与可视化渲染**：子线程成功调取了 `cv::rectangle` 和 `cv::putText`，并在界面弹出了顺滑、不跳帧且不崩溃的 `cv::imshow` 测试窗口。

> **下一步计划（阶段六：边缘网关形态的终极闭环）**：
> 目前 `cv::imshow` 只是用于我们插着显示器做调试。作为一台真正的“网关”服务器，我们要把画好框的 `cv::Mat` 视频流重新压回到队列中，交由 `streamerThread` (推流拼接线程)。并在这里利用 **RGA 硬件拼接**多路流，最后交给 **MPP 硬件编码器 (H.264/H.265)**，把处理后的成片网络发送出去 (RTSP/RTMP)，彻底完成 AI 边缘盒子的闭环！

### 阶段六：网关流媒体化与 MPP 硬件编码输出 (当前完成)
这是作为“AI 边缘网关 (AI-NVR)”最核心的终点，我们要将上面各个线程的心血打包、硬编码并极速送出。
- **全链路队列打通**：新引入了全局后端的 `g_push_queue`，在 `inferenceThread` 把 NPU 画好人脸绿框的帧不落地直接推送给大后方的推流线程。
- **GStreamer 与 MPP 联动硬编**：在 `streamerThread` 我们彻底抛弃了无法在服务器/SSH无显示器上运行的 `cv::imshow`（上一阶段曾导致 GTK 崩溃）以及极其耗费 CPU 的原生软件 `writer`。
- **写入瑞芯微底座**：借助 OpenCV 内部隐式调用的 GStreamer 管道：`appsrc ! videoconvert ! format=NV12 ! mpph264enc ! mp4mux ! filesink`。我们直接调起了 RK3588 的 **mpph264enc 核心硬件编码器**。这使得我们的带画框结果的视频，能够不占用一丁点宝贵的 Linux CPU 算力，利用专门的 VPU 进行 H.264 零延迟压缩！并在本地输出 `gateway_output_hw.mp4` 进行最后验证。一旦验证成功，把 filesink 改成 rtmp/rtsp sink，这就是一台真正的商业级监控分析服务器！
