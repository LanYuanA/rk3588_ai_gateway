# RK3588 多路 AI 视频流网关项目：性能优化与底层排坑实录

这份文档详细记录了在 RK3588 平台上开发“四路 RTSP/V4L2 视频流拉取 -> NPU 目标检测 -> 画面拼接 -> 硬件编码推流”项目时，从严重卡顿、内核死机到最终实现多路满帧丝滑运行的完整技术演进过程。非常适合作为简历的**项目难点攻克**和**性能优化**部分的复盘资料。

---

## 1. 项目初始状态与初步痛点

**初始设计：**
* 使用 OpenCV (`cv::VideoCapture`) 从网络和 USB 摄像头拉取多路视频。
* 使用 `cv::resize` 进行分辨率统一，使用 `cv::hconcat` / `cv::vconcat` 或新建 Mat 进行四宫格拼接。
* 给不同线程之间人为增加 `sleep_for(33ms)` 来试图控制 30FPS 的处理节奏。

**遇到的问题：**
1. **极端卡顿**：多路 1080p 视频拼接后，画面卡顿严重，CPU 占用率爆满。
2. **实时流画面定格**：本地 MP4 测试正常，但拉取网络的 RTSP 流或物理摄像头时，画面基本静止或严重掉帧积压。

## 2. 第一阶段优化：CPU 架构级重构（零拷贝与流控）

### 2.1 零拷贝 (Zero-Copy) ROI 画布拼接优化
* **排查**：原有利用 `clone()` 以及矩阵拼接函数会在内存中进行极其频繁的 Deep Copy，1080p/480p 的图片每秒进行上百次深拷贝，直接导致总线带宽高负荷。
* **操作**：废弃所有深拷贝拼接。在推流线程启动时，在内存中**预分配一块巨大的 1280x960 (2x2格) 空白底布** `cv::Mat final_grid`。
* **效果**：后续图像更新时，利用 `cv::Rect` 获取 ROI 区域引用，采用 `canvas[i].copyTo(roi_mat)` 就地覆盖内存，将拼接时间消耗直接降至毫秒级下限。

### 2.2 实时流休眠剔除机制
* **排查**：网络流（RTSP）与物理摄像头属于**真实实时流**，数据包由远端或硬件自然按帧率送达。如果由于拉流线程内有 `sleep_for(33ms)`，当网络产生微小抖动时，读取窗口就会错过，导致堆积并触发 GStreamer/FFmpeg 的防爆音机制（Drop frames），导致画面看起来“定格”。
* **操作**：增加 `is_live_stream` 判断。判断 URI 包含 `rtsp://` 或 `/dev/` 时，**彻底去除拉流线程的人为 sleep**，依靠网络 Socket 自身的阻塞机制去被动等待画面；仅对瞬间可以读完的本地 `.mp4` 文件保留休眠控制。

## 3. 第二阶段踩坑：追求极致性能导致的 Linux Kernel Panic

为了解决 CPU 软缩放（1080p -> 480p）的性能瓶颈，决定调用 RK3588 独有的 **RGA (2D Raster Graphic Acceleration Interface)** 硬件图形引擎来进行硬件缩放。

### 3.1 “抄作业”引发的系统崩溃
* **现象**：从 Github 克隆了某开源 RGA 测试项目并按其逻辑运行后，**直接导致整个 RK3588 操作系统瞬间黑屏死机重启（触发 Linux 内核 Oops: 96000004）**。
* **根因深度分析**：
  原作者代码为了方便，使用了系统标准的 C 库 `malloc()` 分配了基于分页机制的用户空间虚拟内存，并调用 `importbuffer_virtualaddr` 强行骗过硬件直接喂给 RGA DMA 引擎使用。在多流并发、或者由于 Ubuntu 22 的 IOMMU 页表无法对齐限制下，直接引发了严重的内存越界访问（Memory Access Violation），被 Linux Kernel 在底层拦截由于涉及到显卡引擎，直接内核崩溃（Panic）。

### 3.2 化险为夷的设计兼顾
* 放弃高风险的 RGA 自己分配内存逻辑，转而通过提取官方 RGA 头文件 `im2d_*.h` 接入我们的 Cmake 工程。
* **安全兜底重写**：依然使用 OpenCV 稳定分配的 `Mat.data` 虚拟地址，单张传入 `wrapbuffer_virtualaddr()` 给 RGA 硬件执行 `imresize`。**关键在于加入了回退机制**（`if (ret == IM_STATUS_SUCCESS)`），如果硬件申请处理失败或者页表对齐异常，不至于死机，而是平滑退火走到 `cv::resize` 纯 CPU 运算，完美在性能和系统健壮性间找到了平衡。

## 4. 第三阶段排坑：RTSP 黑屏与依赖污染大屠杀

在引入了 RGA 后，发现所有 `rtsp://` 网络流竟然读不到视频了，但是运行环境相同的 Python cv2 脚本与终端 `ffmpeg` 命令行均能出图。

### 4.1 第三方库环境依赖污染 (Dependency Pollution)
* **排查**：查看 CMake 构建日志发现，引用的 OpenCV 版本被悄悄降级成了 `3.4.5`，而不是系统的 `4.2.0`。
* **根因**：下载第三方 RGA 代码时，其包内偷偷夹带了一个极度阉割（无网络和 FFmpeg 支持）的预编译 opencv 目录系统，导致 CMake 的 `find_package(OpenCV)` “认认贼作父”。
* **解决**：果断清理带有污染的 3rdparty 文件夹，清空 build 缓存，重新 `cmake ..`，强行找回系统级自带完整 FFmpeg 能力的 `/usr` 目录下的 OpenCV 4.2.0。

### 4.2 音频轨道堵塞视频 (Audio Track Backpressure)
* **排查**：查看 MediaMTX 服务端日志，发现该网络流带有 `MPEG-4 Audio` 轨道。
* **根因**：早期为了调用底层的硬件能力使用了 GStreamer 进行拉流，GStreamer 的 `decodebin` 解析出视频和音频双轨，而我们的 C++ 管道并未处理音频节点（Sink）。随着缓冲区打满，几秒钟后彻底憋死视频流（反压），导致 Socket 关闭（EOF）。

### 4.3 终极硬件驱动死锁冲突 (V4L2 M2M Conflict)
* **问题**：将 OpenCV 后端切为 `cv::CAP_FFMPEG` 后，居然在 C++ 代码中依然阻塞黑屏，但同环境 python 正常！
* **深入分析**：RK3588 的系统级 FFmpeg 默认注册了一个硬件解码器 `h264_v4l2m2m`！但是在 C++ OpenCV 中直接调用时，该驱动节点在面对多路并发时极易发生死锁（或者被其他进程如推流的 `mpph264enc` 排轨）。而 Python 便携包因为不带特殊硬件支持所以走软解成功。
* **黄金修复**：在代码拉流前注入极其强硬的环境变量，利用 `video_codec;h264` 强制 FFmpeg 绕过内核驱动 bug，走纯 CPU 软解，同时加入低延迟与 TCP 配置。
  ```cpp
  setenv("OPENCV_FFMPEG_CAPTURE_OPTIONS", "rtsp_transport;tcp|stimeout;2000000|flags;low_delay|video_codec;h264", 1);
  cap.open(stream_url, cv::CAP_FFMPEG);
  ```

## 5. 黄金系统架构（简历亮点总结）

历经重重磨难，该项目最终搭建起了一套**高度异构、针对 RK3588 极限压榨算力的黄金性能架构**：

1. **极其健壮的网络拉取 (Pull)**
   * **FFmpeg 纯 CPU 软解**：利用环境变量彻底屏蔽不稳定的 `v4l2m2m` 硬解驱动，自动过滤干扰音频，利用 CPU 处理纯净 H.264 数据为 BGR 画面，保证不断网不死锁。
2. **极限性能缩放 (Resize)**
   * **RGA 2D 硬件加速引擎**：将软解出来大张的高清图 (1080p)，通过 `librga.so.2` 的 C API 切入硬件底层，进行零 CPU 负担的极限强行缩小。
3. **隔离流转 (Dispatch)**
   * **ThreadSafeQueue 双轨分离队列**：视频帧被克隆两份进入分离式的独立队列（推理队列 / 推流队列），杜绝共享锁导致的数据争抢。NPU 可以按自己的推理耗时“随意扔帧”，而推流端始终取到最新帧。
4. **零拷贝拼接推流 (Merge & Push)**
   * **ROI 内存就地覆盖**：单张1280×960巨型背景画布底布设计，跳过 `cv::hconcat` 深拷贝。
   * **MPP 硬件编码推流**：生成的拼接画面不走 CPU，而是通过 GStreamer `appsrc ! mpph264enc` 调用瑞芯微独立视频硬编单元，切片封包为 MP4 或直推流媒体分发服务器。

**最终成果：** 以极微量的 CPU 算力，稳定实现了 4 路同源并行、防网络抖动、并搭载 NPU 的本地/云端网关，排除了所有硬件 API（内核崩溃级别）的底层炸弹。
